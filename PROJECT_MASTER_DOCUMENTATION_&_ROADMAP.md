## 5. Core Product Principles / Product Philosophy

All future development must follow these principles.

### 5.1 Not a Profit Guarantee System / Accuracy First
TITAN ORACLE must never present itself as a guaranteed-profit machine. Trading decisions must never be based on decorative UI, fake certainty, or unvalidated calculations. Mathematical validity is mandatory. The system may recommend, validate, monitor, protect, or execute under rules, but every trading decision remains subject to risk.

### 5.2 User Ownership / Control by Mode
The user owns: capital, risk tolerance, permissions, execution mode, accepted mission, and final responsibility. The AI system provides intelligence, governance, and protection. Different execution modes have different authority. The system must clearly distinguish advisory mode, rule-based mode, AI-managed protection, and future autonomous AI Auto Pilot.

### 5.3 Capital Protection
AI execution modes must prioritize fund protection, liquidation prevention, and controlled exposure. The system must prioritize capital protection once a trade enters a supervised or AI-managed mode. In AI Copilot Execution and AI Auto Pilot modes, the AI may close or reduce exposure early if real-time market risk becomes dangerous.

### 5.4 Institutional Clarity / Semantic Clarity
Every displayed metric must answer a specific question. No metric should create ambiguity. Terms such as Risk Score, Execution Readiness, Consensus Bias, and Position Allocation must never overlap.
- **Risk Score:** How dangerous is this signal or market condition?
- **Execution Readiness:** Can this trade be executed cleanly right now?
- **Consensus Bias:** How assertive is the multi-AI agreement?
- **Position Allocation:** How much capital should be exposed?
- **Decision Gate:** Should this trade be accepted, reviewed, rejected, or delayed?

### 5.5 Institutional UI Discipline
The UI must reduce cognitive load, avoid meaningless decoration, and show only decision-relevant information. Backend and UI must be separated; UI must render calculated states and not contain core trading math.

### 5.6 Stability Before Expansion
Before adding major new features, existing UI, data flow, and risk logic must be stable.

## 7. Execution System Overview & Mode Definitions

TITAN ORACLE has four major execution-related systems, plus a basic manual-review baseline.

### Execution Hierarchy
| Level | System | Status | Authority |
|---:|---|---|---|
| 0 | Manual Review | Base | User reviews signal manually |
| 1 | AI Copilot Assist Only | Added | AI assists, monitors, explains, warns; no execution |
| 2 | Auto-Trading | Added | User-defined rules execute; AI does not arbitrarily close |
| 3 | AI Copilot Execution | Added | User accepts/inputs trade; AI may protect/close based on danger |
| 4 | AI Auto Pilot | Not added yet | AI scans, selects, supervises, and manages preset short-term trades |

### 7.1 Level 0 — Manual Review
Manual Review is the most basic mode. User only sees analysis. No AI execution, no automated close. 
AI can display: signal data, confidence, risk score, probability, direction, entry/current/expected/predicted values, TP/SL, risk/reward, supporting indicators, and suggested action. 
The user decides everything manually. AI has no execution authority.

### 7.2 Level 1 — TITAN AI Copilot Assist Only
**Status:** Added / intended feature.
**Purpose:** Assistance, guidance, monitoring, explanation. No execution authority.
AI can: explain signals, validate logic, detect contradictions, warn about risk, monitor active trades, recommend actions, show Decision Gate, show Execution Readiness, show Stop-Loss Sanity, show Consensus Disagreement, show Model Provenance, and identify conflicts.
AI cannot: open trade, close trade, modify trade, change allocation, move stop-loss, reduce exposure, or trigger exchange order.
This is advisory-only mode. It is the safest and should be available to all users. Best used for learning, reviewing, manual trading, validating trade logic, and observing market conditions.

### 7.3 Level 2 — Auto-Trading
**Status:** Added / intended feature.
**Purpose:** User-defined rule-based execution / deterministic automation.
In Auto-Trading, the user manually inputs trade conditions. The system follows those conditions deterministically.
**Core rule:** Trade will not close until all user-defined valid conditions are filled.
Typical user inputs: pair, direction, entry, take-profit condition, stop-loss condition, close rule, investment amount, optional validity window, timeframe condition.
AI role: optional monitoring, optional warning, optional validation. AI does not make discretionary close decisions in this mode unless a future optional emergency circuit breaker is explicitly enabled by user.
*Important distinction:* Auto-Trading is not AI Auto Pilot. Auto-Trading follows user rules. AI Auto Pilot selects and manages under preset AI rules.
Recommended safety extension for later: emergency exchange desync lock, API failure alert, liquidation emergency alert, invalid market state pause.

### 7.4 Level 3 — TITAN AI Copilot Execution
**Status:** Added / intended feature.
**Purpose:** User-approved trade with AI-managed protection.
In this mode, the user manually inputs a trade or accepts AI-recommended values (e.g., from Signal Pro or Live Radar). After the trade starts and enters Mission Center, AI monitors real-time market conditions and can take protective action.
AI can: close trade early, reduce exposure, tighten exit logic, protect capital, prevent liquidation, react to extreme volatility, abort if market flips, warn user, trigger emergency close if authorized, move mission to protection state, and override holding target for fund safety.
AI cannot: randomly trade without user-approved entry or accepted setup, ignore hard risk rules, or violate exchange/account limits.
**Core rule:** In AI Copilot Execution, capital preservation is allowed to override target-holding if real-time market danger becomes severe.
This mode is especially important for futures, volatile assets, and fast market regime changes.

### 7.5 Level 4 — TITAN AI Auto Pilot
**Status:** Not added yet.
**Purpose:** Short-term AI-managed opportunity scanning and preset execution.
AI Auto Pilot will live primarily inside the Live Radar module. It is a future autonomous short-term system. It will behave like a guarded AI trading bot, but not an uncontrolled bot.
Core concept:
- User enters AI Auto Pilot area ("Scan for AI Auto Pilot").
- User selects limited preset scope (No deep manual customization initially).
- System scans for short-term high-probability moves.
- If all hard gates pass, system accepts trade.
- User is moved directly to Mission Center, which supervises the specific trade.
- AI manages profit shield and emergency exits.

## 8. AI Auto Pilot — Short-Term Mode Specification
### 8.1 Location

Entry label:

### 8.2 Initial User Options
Keep the first version controlled and non-customizable (no custom free-form setup in first release).

Investment presets: IN1 = $10, IN2 = $15, IN3 = $30, IN4 = $50.

### 8.3 AI Auto Pilot Objective
AI Auto Pilot scans for the most probable large movement in a short period.
Target behavior / Goal:
 * take profit automatically
 * protect principal progressively / lock profit progressively
 * take controlled small exposure
 * reduce exposure as profit increases / trade moves favorably
 * close early during danger / if market becomes dangerous
 * supervise trade in Mission Center
 * avoid full liquidation impact / full fund liquidation impact
### 8.4 Original Milestone Criteria
The owner’s original target states remain the strategic criteria:
State-1: +3%, State-2: +7%, State-3: +10%, State-4: +12% / 12.5%, State-5: +15%.
### 8.5 Profit Accounting Rule / Profit Shield Ladder
The original user intention is preserved: the system should support profit targets while progressively saving investment. However, values must be separated.
 * **Projected Profit:** Profit over the full original investment if the whole position reached that ROI (e.g., $50 × 7% = $3.50).
 * **Locked / Realized Profit:** Actual realized profit from the portion of the position that was closed (e.g., Closed capital $12.50 × 7% = $0.875).
 * **Remaining Exposure:** Capital still exposed after partial close.
 * **Safe Zone State:** Level of capital protection achieved.
   Mission Center must show these when useful.
### 8.6 Recommended Trial Ladder — Balanced Profit Shield
This is the recommended first implementation ladder because it protects capital while still allowing a meaningful runner.
For IN4 = $50:
| State | ROI Trigger | Action | Closed Capital | Locked Profit | Remaining Exposure |
|---|---|---|---|---|---|
| State-1 | +3% | First safety lock / profit confirmation | 10% = $5.00 | $0.15 | $45.00 |
| State-2 | +7% | Safe Zone-1 (reduce exposure) | 20% = $10.00 | $0.70 | $35.00 |
| State-3 | +10% | Safe Zone-2 (secure stronger profit) | 25% = $12.50 | $1.25 | $22.50 |
| State-4 | +12.5% | Safe Zone-3 (preserve gain) | 25% = $12.50 | $1.5625 | $10.00 |
| State-5 | +15% | Final close (complete trade) | remaining 20% = $10.00 | $1.50 | $0.00 |
Approximate total locked profit: $5.1625. This is less than holding the full $50 to +15% ($7.50), but it provides far better protection / reduces exposure along the way.
**Why This Ladder Exists:** The ladder exists because no risk means no gain, but unmanaged risk can destroy the user’s capital. The system allows the trade to breathe, but after the market proves movement, it gradually removes exposure. This means: user can capture profit, trade can run, profit is locked early, sudden reversal damage is reduced, and liquidation impact is less severe.
### 8.7 Internal Adaptive Modes Later
The user should not manually select these initially; AI selects internally.
 * **Protect Mode:** Used in volatile conditions. Closes more early, lower max profit, higher capital safety.
 * **Balanced Mode:** Default trial logic. Follows state ladder above.
 * **Runner Mode:** Used only in very strong conditions / when signal quality, liquidity, and consensus are exceptionally strong. Closes less early, allows more position to run, higher profit potential, still has emergency exit.
### 8.8 AI Auto Pilot Hard Gates / Rejection Gates
AI Auto Pilot must reject trade if any hard gate fails:
 1. Signal direction invalid.
 2. Stop-loss direction invalid.
 3. Risk/reward sequence invalid.
 4. Risk Score is EXTREME.
 5. Execution Readiness is POOR.
 6. Data Freshness stale / not active.
 7. Spread unacceptable.
 8. Slippage unacceptable.
 9. Liquidity insufficient.
 10. Consensus Confidence below minimum/threshold.
 11. Consensus Disagreement too high.
 12. Consensus Bias conflicting with danger state.
 13. Portfolio exposure exceeds limit.
 14. Validity Window expired/not active.
 15. Oracle/data desync occurs.
 16. Hard conflict flag is severe.
 17. Exchange/API state unavailable.
### 8.9 Early Exit Conditions
From +1% to +12.5%, trade may be closed anytime for investment safety if real-time market conditions become dangerous. AI may close early if: Risk Score becomes EXTREME, Execution Readiness POOR, spread widens, liquidity collapses, slippage spikes, orderbook reverses sharply, BTC/market regime flips, derivatives/funding/OI stress spikes, liquidation risk increases, stop-loss becomes invalid, oracle desync occurs, or Mission Center detects capital-protection threat.

## 10. Institutional Semantic Architecture (Source-of-Truth)
The following terms are mandatory and must not be mixed. The project must use strict semantic separation.
### 10.1 Risk Score
**Meaning:** Objective danger score/level of the signal or market condition.
**Values/Ranges:** 0–15 = LOW, 16–25 = MEDIUM, 26–35 = HIGH, 36–100 = EXTREME.
**Bangla:** RISK SCORE = ঝুঁকির পরিমান.
### 10.2 Execution Readiness
**Meaning:** Mechanical ability to execute the trade cleanly safely at this moment.
**Inputs:** spread, liquidity, slippage, data freshness, orderbook state/depth, latency, execution friction, API availability.
**Values:** OPTIMAL, ACCEPTABLE, DEGRADED, POOR. Recommended usage in Signal Pro, Live Radar, Decision Gate, Mission Center.
### 10.3 Consensus Bias
**Meaning:** AI consensus posture / assertiveness. This does not mean danger. It means AI conviction posture.
**Values:** CONSERVATIVE (Cyan), MODERATE (Green), AGGRESSIVE (Gold). Never use red for Aggressive.
### 10.4 Position Allocation
**Meaning:** Capital sizing recommendation.
**Values:** Conservative = 2.0% Cap, Moderate = 5.0% Cap, Aggressive = 10.0% Max.
This must synthesize: Risk Score, Execution Readiness, Consensus Bias, portfolio exposure, trade/permission mode, market regime.
### 10.5 Forbidden Confusion / Correct Example
Do not display: RISK PROFILE = AGGRESSIVE. Use CONSENSUS BIAS = AGGRESSIVE. For danger, use RISK SCORE = LOW/MEDIUM/HIGH/EXTREME.
**Correct Example:** Consensus Bias: AGGRESSIVE, Risk Score: EXTREME, Execution Readiness: DEGRADED, Position Allocation: Conservative 2.0% Cap, Decision Gate: CAUTION. This means AI sees strong opportunity, but market danger is high, so capital exposure must be limited.

## 12. Mathematical Validation Rules
### 12.1 Stop-Loss Direction
**Mandatory:**
LONG / SPOT: StopLoss < EntryPrice
SHORT: StopLoss > EntryPrice
If violated, state is INVALID SIGNAL STATE or LOGIC FAULT — stop-loss violates trade direction. Normal signal should not render / block normal rendering.
### 12.2 Risk/Reward Direction Sequence & Formula
LONG: Target > Entry > StopLoss
SHORT: StopLoss > Entry > Target
**Formula:** Risk = abs(Entry - StopLoss), Reward = abs(Target - Entry), RR = Reward / Risk. If invalid, block normal rendering.
### 12.3 Stop-Loss Sanity Label
Add contextual SL distance thresholds:
 * <= 5%: Standard
 * > 5–10%: Elevated
 * > 10%: Extreme / Review Sizing

**UI Architecture Principle:** UI should behave like a terminal renderer / dumb terminal. Composable functions should render state, display labels/colors, and handle interaction. They should not calculate stop-loss validity, risk/reward, risk score, consensus disagreement, trade closure, or contain business logic if avoidable.
**Compose Performance Rules:** Use tabular numerals (fontFeatureSettings = "tnum") for all market numbers (price, percent, timer, score, ROI, TP/SL, rank, latency, volume, spread, slippage). Use maxLines = 1 and overflow = TextOverflow.Ellipsis for compact labels, but avoid clipping critical values like CONSERVATIVE.
**Animation Rules:** Avoid heavy cinematic animations, layout-jumping slide transitions on live market rows, proportional numerical fonts, and raw loading placeholders. Use stable transitions.
